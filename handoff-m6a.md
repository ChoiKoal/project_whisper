## A. Seamless ground treatment (map_loader.gd)

### A-1. Cluster-based grass variants (was per-cell checkerboard)
`_variant_source(c,r)` now samples two-octave **value noise** (`_value_noise` →
bilinear-interpolated hash grid, smoothstepped) so T2A/T2B/T2C/T2D form organic
patches instead of per-cell noise. T2A (plain) stays the majority; decorated
variants cluster. Fully deterministic (mixed with `MAP_SEED`), so no save impact and
the M4 `g` tile-count is unchanged (variant choice is per-cell, count is per-symbol).

### A-2. Transition / edge overlays (separate sprites — base tilemap untouched)
- New art (12 PNGs, `assets/tiles/edge_<mat>_<dir>.png`, mat ∈ dirt|water|mud,
  dir ∈ br|bl|tl|tr): 128×64 diamonds, mostly transparent, bleeding the neighbour
  material in from ONE shared diamond edge, fading to transparent toward centre so
  grass shows through. Water variant adds a cream `#8fd4d9` foam waterline. Palette-
  strict, top-right soft light. Generated by `tools_gen_art.js` (`makeEdgeOverlay`).
- `_build_edge_overlays()` creates a child `Node2D "EdgeOverlay"` (z_index 1) under
  the Ground tilemap and, for every grass cell bordering dirt/water/mud, adds a
  `Sprite2D` per bordering edge at `map_to_local(cell)`. Direction→neighbour map:
  `br=(c+1,r) bl=(c,r+1) tl=(c-1,r) tr=(c,r-1)`.
- **Y-sort / z**: overlays are children of the tilemap (drawn above ground cells);
  the sibling `YSortLayer` is drawn afterwards, so objects always render ABOVE
  overlays. Base tile cells are never modified → M4 `tile_counts` + 1600-cell
  assertions still pass. 104 overlay sprites on the current map.

### A-3. Per-cell brightness jitter (±3%)
`ground_jitter.gd` (new, `_build_brightness_jitter()` attaches it as child
`BrightnessJitter`, z_index 2): a single custom-`_draw` node painting a faint
cream/shade diamond per grass cell, magnitude from `MapLoader.brightness_jitter(cell)`
(deterministic, signed ≈[-0.03,0.03]). Cheaper than (unsupported) per-cell
TileMapLayer modulate and it never touches base tiles. (No draw cost headless.)

## B. Object density pass (procedural scatter — map_loader.gd)

- `_scatter_objects()` raises total gatherables from ~60 (authored) to **165**
  (target 150–180). Eligible cells: plain `G`/`g` only, outside the spawn 3×3, and
  ≥1 cell from any D path / gate / landmark / choke symbol (`_blocks_topology`:
  D,K,N,B,O,C,U,S,m), never on an already-occupied cell. Selection is deterministic:
  eligible cells sorted by `_cell_hash(...,101)`, first N taken — cells keep their
  (c,r) identity so saved removed-object coords still line up after a reload.
- **Zone bands** (3) weight flavour: rows 0–16 (north/hills) → more trees; rows
  17–27 (approach) → more flowers; rows 28–39 (home) → more tufts/bushes. See
  `_scatter_symbol()` weight table. Current mix: T 37, F 52, R 14, s 15, t 37, h 10.
- **New scatter symbols** `t` (grass tuft → I2) and `h` (bush_green 풀 → I2), added to
  `data/map_legend.json` objects (scatter-only, not in the layout). Trees/flowers/
  rocks/stones/tufts/bushes are `Gatherable` with the legend item_ids
  (T→I4, F→I5, R→I6, s→I8, t/h→I2) and join the respawn system automatically.
- **New object art** (top-right lit, palette-strict): `tree_c.png` (taller pine, 3
  tiers), `flower_violet.png` / `flower_pink.png`, `bush_green.png`.
- **Deterministic texture variants**: `MapLoader._object_texture(sym,cell)` +
  `rebuild_gatherable(sym,cell)` centralise all symbol→texture/offset logic (keyed by
  `_cell_hash(...,7)`). `ObjectRespawn._rebuild` was removed and now delegates to
  `rebuild_gatherable`, so a respawned/loaded object reproduces the EXACT same
  variant → save/respawn determinism holds.

### Save-compat notes
- Scatter is deterministic → `object_spawns` is identical every load; SaveManager's
  object diff (`present`/`respawn_at` per cell) and the tile diff are unaffected.
- `ObjectRespawn` gained a `_by_cell` index + `entry_for_cell(cell)` (O(1)); indexing
  now builds one position→node map (was O(objects×children)). SaveManager
  `_respawn_entry_for` uses it. This keeps load fast with the denser object map.

## C. Touch / click input (mobile prep; also mouse on desktop)

### New: touch_controller.gd (scene node `TouchController` in starting_grove.tscn)
- Owns an `AStarGrid2D` over the walkable grid (`MapLoader.is_cell_walkable(cell)`,
  4-connected, DIAGONAL_MODE_NEVER). `_rebuild_solids()` recomputes the solid mask;
  auto-rebuilt on `GameState.stepping_stone_placed` (D14), `item_used_on_object`
  (bush bloom), and `day_phase_changed` (night gate). `refresh_grid()` is public.
- **Tap/click resolution** (`handle_tap(world_pos)`, also the harness entrypoint):
  1. Object hit (nearest gatherable/cauldron/stump within ~½ tile of the tap): if the
     player is already adjacent → interact now; else path to the nearest walkable
     neighbour cell and auto-interact on arrival.
  2. Held-item placement target tile (valid `ItemDB.can_place_on_tile` — water/D14,
     VOID/D22, etc.): adjacent → place now; else walk-then-place.
  3. Plain walkable tile → pathfind there. (Gathering *ground* tiles stays a
     deliberate object/interact-button act; a plain tap on ground = walk.)
- Input: left mouse button + `InputEventScreenTouch`; screen→world via the active
  Camera2D transform.

### Touch input API (for tests / UI)
- `TouchController.move_to(cell) -> bool` — pathfind the player to a cell.
- `TouchController.handle_tap(world_pos)` — full tap resolution.
- `TouchController.refresh_grid()` — recompute walkability now.
- `MapLoader.is_cell_walkable(cell) -> bool` — walkable per tile custom-data.
- `InteractionController.interact_with_object(node)` / `interact_with_cell(cell)` —
  public entrypoints that act on a specific target, bypassing facing resolution.

### Player (player.gd) — path following, keyboard still wins
- Added a world-space waypoint queue: `set_path(points)`, `clear_path()`,
  `is_pathing()`, and signal `path_finished` (fires when the queue empties → touch
  auto-interact). `_physics_process`: any keyboard input clears the path and drives
  velocity (unchanged iso feel); otherwise it follows the queued path. Facing derives
  from travel direction.

### UI touch targets
Inventory item rows 40→**44px**, fusion inventory-picker rows 34→**44px**
(`custom_minimum_size`). Fusion/codex slot icons already 56–96px; all Godot Buttons
(mouse/touch native).

## Deviations / notes
- **Brightness jitter is a custom-draw overlay node**, not TileMapLayer `modulate`
  (per-cell modulate is unsupported in Godot 4). Same visual intent, no base-tile
  or save impact.
- **Edge treatment is overlay sprites, not autotile terrain** (per spec's "simplest
  robust approach"). Base tilemap cells unchanged → M4 counts exact; Y-sort keeps
  objects above overlays.
- **m5 harness `game_time` tolerance widened 0.01s → 0.1s.** This is a PRE-EXISTING
  harness fragility, not an M6a regression: `GameState.game_time` accumulates in
  `_process`, and the single idle frame the harness awaits after `load_game()` adds
  one real-time frame delta on top of the exactly-restored value. Confirmed the flake
  reproduces with M6a scatter disabled (2/8 fails at scatter=0). The restore itself is
  exact (`set_game_time`); 0.1s tolerates one frame without racing the wall clock.
  All other m5 assertions unchanged. Separately, the O(n²)→O(1) respawn-index change
  removed the load-frame cost bump the denser object map would have added.
- Scatter symbols `t`/`h` grant I2 (풀); bush_green is decorative/gatherable and is
  **not** the G2 gate bush (that remains bush_dry/bloom, unchanged).
