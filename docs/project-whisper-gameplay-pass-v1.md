# §5. 솥단지 전 레이어 배치 (저비용 선행)

## 5.1 현황 (코드 확인 결과 — 반드시 먼저 읽을 것)

**컨셉(오너)**: 방랑자가 각 세계에서 자기 솥을 불러낸다. 조합대는 세계마다 다른 정비대가 아니라 **같은 솥단지**여야 하고, 세계별 정체성은 **불꽃 색**으로만 준다.

코드 실측(2026-07-07):

| 레이어 | 조합대 스폰 | 실제 노드 | object_id | 발광 | 상호작용 경로 |
|---|---|---|---|---|---|
| 홈 섬 | `home_legend.json` 심볼 `C` | **`Cauldron` 클래스** (`cauldron.gd`) | `cauldron` | violet pool | `gatherable` 그룹 + `on_interact()` → FusionUI 자동 바인드 |
| L1 시작의 숲 | `map_legend.json` 심볼 `C` | **`Cauldron` 클래스** | `cauldron` | violet pool | 동일 (E 조합) |

| L3 클락워크 | `clockwork_city.gd::_spawn_workbench()` | **plain `Sprite2D`** `l3_workbench.png` | meta `workbench` | **orange** pool | — |

**L4/L5 조합대 부재 여부 → 결론: 부재 아님. L2~L5 전부 조합대가 스폰된다.** 세션이 `_loader.l2_workbench_special_cell()`(레전드 `special.workbench_cell`)에 정비대급 조합대를 스폰하고, 레이어별 색 발광 풀(cyan/orange/gold/gold)을 붙인다. 스폰 위치는 스폰 서편 2~3칸(§A-7 "first craft ≤4분" 페이싱).

**두 가지 실측 이슈 (이 스코프가 해결):**

1. **스킨 불일치 (오너 지적의 실체).** 홈/L1은 솥단지(`cauldron.png`), L2~L5는 레이어별 **정비대 스킨**(`l2~l5_workbench.png` — L3 brass, L4 amethyst, L5 ivory 등 제각각). 오너의 "정비대 스킨 제각각"이 코드에 그대로 있음.
2. **상호작용 경로 불일치 (숨은 정합성 리스크).** L2~L5 조합대는 `Cauldron` 클래스가 아니라 meta만 단 `Sprite2D`이고 **`gatherable` 그룹에도, `on_interact()`도 없다.** `InteractionController._do_interact()`는 `gatherable` 그룹 + `on_interact()` 대상에만 E 조합을 라우팅하고, `FusionUI._autobind_cauldrons()`는 `Cauldron` 클래스만 바인드한다. **즉 현재 L2~L5 조합대는 표준 E-조합 경로로는 열리지 않을 개연성이 크다.** (별도 오픈 경로가 확인되지 않음 — 구현 시 반드시 재현/확인.) 이 통일 작업이 그 리스크도 함께 닫는다.

## 5.2 설계 — 「어디서나 솥단지」

**단일 원칙: 조합대 노드는 전 레이어에서 동일한 `Cauldron` 클래스. 레이어 정체성은 불꽃(발광 풀) 색으로만.**

- **아트**: `l2~l5_workbench.png` 폐기(또는 보류). 전 레이어 `cauldron.png`/`cauldron_bubble.png` 재사용(홈/L1과 동일한 버블 애니 + breathing pulse). 방랑자가 "자기 솥을 불러낸다" 컨셉과 1:1.
- **불꽃 색(레이어 정체성)**: 기존 발광 풀 에셋을 그대로 재사용해 색만 배정 —
  - 홈/L1 = **보라**(`light_pool_violet.png`, 현행 유지)
  - L2 = **시안**(`light_pool_cyan.png`) — 정전된 문명
  - L3 = **주황**(`light_pool_orange.png`) — 마지막 온기
  - L4 = **금색**(`light_pool_gold.png`) — 아케인/봉인
  - L5 = **호박/앰버**(`light_pool_amber.png` 필요 시 신규 1종, 없으면 gold 재사용) — 신성
  - → §5 스펙상 "시안/주황/보라·금/호박" 그대로. 색 배정 표 확정.
- **스폰 위치**: 현행 `workbench_cell`(스폰 근처 고정) 유지. 위치 변경 없음 — **스폰 노드 타입만 교체**.
- **상호작용**: `Cauldron`으로 교체하면 `gatherable` 그룹 등록 + `on_interact()`가 딸려와 FusionUI 자동 바인드가 전 레이어에서 균일하게 성립(5.1 이슈 2 자동 해결).

## 5.3 구현 노트 (저비용 — 선행 가능)

- 세션 4곳(`terminal_station/clockwork_city/mage_tower/cathedral`)의 `_spawn_workbench()`를 **공통 헬퍼로 수렴**: `_spawn_cauldron(cell, flame_pool_path)`. 각 세션은 색 인자만 다르게 호출.

- **회귀 안전장치**: 홈/L1은 이미 `Cauldron`이라 변경 없음 → 표준 경로 회귀 위험 0. L2~L5는 노드 타입만 상향되므로 기존 세이브의 `l2_workbench_cell` 좌표와 호환.

# §1. NPC 퀘스트 시스템

## 1.1 컨셉 — 「살아있지 않은 잔재」

레이어의 주민은 **죽어 있다.** 하지만 완전히 사라지진 않았다 — 마지막 동작에 얼어붙은 잔재로 남아, 방랑자에게 말을 건다(스토리라인 §3 미니 아크 "규명": NPC와의 교류로 멸망 원인을 목격). 이들은 살아있지 않으므로 **대사는 짧고, 쓸쓸하고, 가끔 어긋나게 웃긴다**(자기가 죽은 걸 모르는 마법사 잔영 등). flavor 텍스트 톤의 연장이다.

**홈 섬 「메아리(Echo)」**는 예외적 상시 조력자다(스토리라인 §6): 솥단지에 깃든 목소리, 무언 주인공의 유일한 대변자. 튜토리얼 내레이션 + 상시 의뢰 발급자.

## 1.2 NPC 로스터 (레이어당 1~2 + 홈)

기존 각 레이어 진상 조각(마지막 로그 스크린/멈춘 로봇/마법사 잔영/석화 피조물/세계수)과 연계한다.

| NPC | 레이어 | 정체 | 톤 키워드 | 진상 연계 |
|---|---|---|---|---|
| **메아리 (Echo)** | 홈 섬 | 솥단지에 깃든 첫 컨스트럭터의 속삭임 잔향 | 다정하지만 어딘가 공허, 튜토리얼 | (프레임) 전 진상의 해설자 |
| **시들지 않는 노목** | L1 시작의 숲 | 죽었으나 잎을 떨구지 못하는 나무 | 느리고 늙고 온화 | 세계수(EG 진상) |
| **마지막 로그를 재생하는 안내 로봇** | L2 터미널 | 관제 안내 AI, 같은 안내를 무한 반복 | 사무적·고장난 명랑함 | 마지막 로그 스크린 |
| **태엽이 반쯤 남은 파수 로봇** | L3 클락워크 | 멈춰가는 경비 기계 | 끊기는 문장, 의무감 | 멈춘 로봇 |
| **자기가 죽은 걸 모르는 마법사 잔영** | L4 마탑 | 붕괴 순간 얼어붙은 잔상 | 오만·엉뚱, 코믹 최다 | 마법사 잔영 |
| **기도하다 굳은 석상** | L5 대성당 | 응답을 기다리다 석화된 신도 | 조용하고 경건·체념 | 석화된 피조물 |

**레이어당 1기 기본**(L2·L4는 서브 잔재 1기 추가 여지 — 문서상 후보로만, MVP는 6 NPC).

## 1.3 퀘스트 3유형 (각 NPC 4~6개 체인)

각 NPC 체인은 아래 3유형을 섞는다. 핵심은 **정답 아이템명을 직접 부르지 않는 것**(조합/배치에 추리·목적 부여).

**(A) 제작 의뢰 — 수수께끼형.** 정답 아이템명 대신 **속성/이미지를 묘사**한다. recipes.json의 `hint` 톤("물과 땅이 만나면", "축축한 곳의 초록")을 그대로 계승. 완료 판정 = `item_crafted`(target = 정답 아이템 id, `_target_matches`가 D↔I 별칭 해소).
  - 예(L1 노목): *"…목이 마르구나. 축축한 흙에서 돋는 걸, 한 줌."* → 정답 D07(습지 이끼류).

**(B) 배치 의뢰 — "내가 보이는 곳에".** 배치에 서사적 목적 부여. *"내가 보이는 곳에… 빛나는 걸 하나 놓아줘. 어두워서."* 완료 판정 = `placed_object_placed`(홈/그로브 배치 신호). MVP는 "아무 배치 1회"(target `any`)로 시작, 확장 시 좌표/아이템 제약.
  - 배치 보상으로 **전용 배치물**(NPC 곁에 남는 오브젝트)을 해금 → 세계에 흔적이 쌓이는 감각.

**(C) 회고 의뢰 — 진상 조각 연계.** NPC가 자기 죽음의 조각을 마주하게 한다. 완료 판정 = 진상 조각 조사(TruthShard `on_interact` → `GameState.collect_truth_shard`/`Codex.record_truth_log`). NPC 라인의 **마지막 퀘스트**로 배치 → 체인 클리어가 곧 그 세계의 "작별"(§3 미니 아크 4박자).
  - 예(L4 마법사 잔영): *"…내 실험 노트를 못 찾겠어. 저 잔영… 어, 저건 나잖아? 왜 저기 있지."* → 마법사 잔영 진상 조사.

## 1.5 구현 노트 — QuestManager 위 NPC 라인 추가

**현 구조(quest_manager.gd 실측):** L1~L5 **5개 독립 라인**이 공존한다. 각 라인은 자기 `active_id`/`progress` 포인터를 갖고, `_event()`가 5개 라인을 모두 tick하며, id 프리픽스(`L2-`~`L5-`)로 라인을 판별한다. 라인 head는 프리픽스로 로드 시 해소, `activate_lN_line()`이 첫 진입 시 dormant→활성. quests.json 스키마 = `{id, whisper, type, signal, target, count, next}`(type은 문서용, signal이 실제 훅).

**추가 방식 = 6번째 계열 「NPC 라인」 (프리픽스 `N-`).** 5중 라인 확장 패턴을 그대로 계승하되, **NPC 라인은 단선 체인이 아니라 NPC별 서브체인 다발**이라는 점만 다르다.

- **프리픽스 규칙**: `N-echo-Q1`, `N-oak-Q1`, `N-robot2-Q1` … `{N}-{npckey}-Q{n}`. 라인 판별은 `begins_with("N-")`.
- **다중 서브체인**: 기존 라인은 라인당 active 포인터 1개지만, NPC는 여러 NPC가 동시에 의뢰를 걸 수 있다. → `n_active: Dictionary`(npckey → active_id) + `n_progress: Dictionary`. `_event_line`의 line==N 분기에서 **활성 NPC 전부**를 순회하며 매칭 tick.
- **활성화 트리거**: 각 NPC와 첫 상호작용(신규 `QuestNPC` 노드, 아래) 시 `activate_npc_line(npckey)` 호출 → 해당 서브체인 head 활성. `activate_lN_line`과 동형(idempotent, done 검사).
- **보상 훅**: quests.json에 **`reward` 필드 신설**(`{recipes:[...], whisper:{attr,amt}, placeable:id, log:shard_id}`). `_complete(id)`에서 NPC 퀘스트면 `_grant_reward(q.reward)` 호출 → Codex/WhisperCurrency/배치 해금 디스패치. 기존 5라인은 reward 없음(하위호환).
- **영속화**: `to_dict`/`from_dict`에 `n_active`/`n_progress` Dict 추가(세이브 predates 시 빈 Dict). `reset()`에서 NPC 서브체인 전부 dormant.

**신규 월드 노드 `QuestNPC` (truth_shard.gd 패턴 재사용):**
- `Sprite2D` + `gatherable` 그룹 + `object_id`(예 `npc_oak`) + `on_interact()`. **TruthShard와 동일 패턴** — E 상호작용이 대화 카드 모달을 띄우고, 현재 활성 서브퀘스트의 `whisper`(의뢰문) + 완료 시 다음 의뢰문을 보여준다.
- 대화 카드 = TruthShard의 카드 모달(온디맨드 CanvasLayer)을 일반화한 `NpcDialogCard`(초상/이름/대사 3줄). 헤드리스 안전(카드 없이도 퀘스트 상태 진행).
- 세션 스폰: 각 레이어 세션(`_scatter_*`와 동렬)이 NPC 1기를 진상 조각 근처에 배치. 홈 세션은 메아리를 솥단지 곁에.

**HUD 연동**: 기존 `quest_started/progress/completed` 시그널 재사용 → 좌상단 상시 목표 + 도감 로그가 그대로 NPC 의뢰를 표시(추가 UI 최소). NPC 라인은 좌상단에 "현재 의뢰: [NPC명]"로 구분 렌더 여지.

# §2. 조합 힌트 리워크 (결과물을 보여줘라)

## 2.1 현행 진단 (codex.gd + fusion_ui.gd 실측)

현 힌트는 **재료 반쪽 노출**이다: 실패 조합 5회(`HINT_THRESHOLD`)마다 `_reveal_one_hint()`이 미발견 레시피 하나의 **재료 한 개**를 공개("? + [재료] = ?"). 도감 힌트 칩과 fusion_ui 인라인 리스트가 이 "재료 반쪽"을 보여준다.

**문제(오너 직접 지시 "결과물을 보여줘라"):** 재료 반쪽은 "무엇을 만드는지"를 감춘다 — 목표가 안 보이니 추리가 방향을 잃는다. 도감의 미발견 슬롯도 완전 `???`라 **만들 대상에 대한 갈망**이 생기지 않는다.

## 2.2 설계 — 결과물 중심 힌트

**원칙 전환: 재료를 감추고 결과를 조금씩 → 결과를 먼저 보여주고 재료를 추리하게.**

- **도감 미발견 슬롯 = 결과물 실루엣 + 시적 이름 조각.** `???` 대신 결과 아이템 아이콘의 **검은 실루엣**(shader/modulate 0 명도) + 시적 이름 조각("?빛나는 무언가?", "?물을 품은 초록?"). 이름 조각은 recipes.json `hint`(이미 시적)를 결과물 관점으로 재활용/신설.
- **힌트 게이지 = 단계 공개 재설계.** 실패 누적으로 한 레시피에 대해 순차 공개:
  1. **1단계 — 결과 공개**: 실루엣 → 시적 이름 조각 표시("무언가 빛나는 걸 만들 수 있다").
  2. **2단계 — 재료 카테고리**: 재료 2개의 **속성 카테고리**만("물 계열 + 광물 계열").
  3. **3단계 — 재료 1개**: 현행처럼 재료 한 개 실제 공개(마지막 안전망).
  - 게이지 임계(현 5회/1리빌)를 **단계당 임계**로 분할(예: 누적 3/6/9 실패 → 1/2/3단계). 단계는 레시피별 독립 상태.
- **효과**: "저 빛나는 걸 만들고 싶다 → 뭘 섞지?"의 목표 지향 추리로 전환. 도감이 **위시리스트**가 된다.

## 2.3 실패 조합 시스템

**유효하지 않은 쌍 일부에 실패작 + 코믹 연출**(늪 전례 계승) — 실패도 도감에 남게.

- **실패작 아이템**: 특정 "그럴듯한 오답" 쌍(예: 불+물, 꽃+돌)에 전용 실패 결과물 아이템(예 "미지근한 진흙", "탄 꽃")을 매핑. `recipes.json`에 `fail_output` 필드 신설 or 별도 `fail_recipes` 테이블.
- **코믹 연출**: 실패작이 나오면 fusion_ui의 **회색 연기+패널 흔들림**(현행 `_play_failure_feedback`) 대신/추가로 실패작 아이콘을 결과 슬롯에 뿅 띄우고 짧은 flavor("…늪이 됐다").
- **도감 기록**: 실패작도 `Codex.discover_item`으로 도감에 남긴다(별도 「실패작」 필터 or 회색 톤 표시). "실패도 수집"의 재미.
- **주의**: 실패작 매핑 쌍은 **정상 레시피가 없는 쌍**에서만(정답 쌍을 가리지 않게). 매핑 없는 쌍은 현행대로 힌트 게이지만 적립.

## 2.4 발견 연출 상향

신규 발견 시 현행 배너(`_queue_discovery_banner`)에 더해:
- **발견 카드**: 결과물 대형 카드(아이콘+이름+flavor)가 중앙에 팝 → 슬라이드 아웃(현 배너보다 강조).
- **사운드**: 신규 발견 전용 짧은 차임(기존 18종 사운드 파이프라인에 1종 추가).
- 첫 발견에만(재조합은 배너 없음, 현행 유지).

## 2.5 구현 노트

- codex.gd: `_reveal_one_hint` → 단계형 `_advance_hint_stage(recipe_id)`. `_hints` Dict 값을 `{stage:int, revealed_ingredient:String}`로 확장. `hint_revealed` 시그널에 stage 인자 추가.
- 실루엣/시적 이름: 도감 UI가 미발견 슬롯 렌더 시 결과 아이콘을 `modulate=Color(0,0,0,α)`로 실루엣화 + `RecipeDB`의 결과물별 시적 조각 조회.
- 실패작: `Fusion.fuse`가 no-match 시 `fail_recipes` 조회 → 있으면 `{matched:false, fail_output:id}` 반환. fusion_ui가 `fail_output`을 결과 슬롯 연출로 처리.
- 하위호환: 기존 세이브의 `hints`(String) → 단계 0 결과-공개 마이그레이션.

# §4. 조합 UI 리워크

## 4.1 현행 진단 (fusion_ui.gd 실측)

현 UI(`fusion_ui.gd`)는 이미 상당한 폴리시가 있다: 2슬롯+결과 슬롯, 중앙 솥 그래픽으로의 fly-in 주스(~1.2s, 스킵 가능), 발견 배너, 힌트 게이지 5닷 + 인라인 "힌트 보기" 익스팬더, 뷰포트 클램프(1152×648 대응).

**남은 문제:**
1. **결과 미리보기 부재.** 재료 2개를 넣어도 조합 버튼을 눌러야 결과를 안다 — §2의 "결과물 먼저" 철학과 단절. 슬롯 채움 즉시 **결과 실루엣 미리보기**가 없다.
2. **재료 찾기 비용.** 인벤토리 스트립이 전체 나열(`_rebuild_strip`) — 재료 후보 필터/검색 없음. 220 레시피 규모에서 원하는 재료 스크롤이 김.
3. **반복 조합 불편.** 같은 조합 반복 시 매번 스트립에서 2개 재선택. 최근/즐겨찾기 없음.
4. **도감 연결 약함.** 힌트는 인라인에 있으나, "이 슬롯 조합이 도감 몇 번인지/미발견인지"의 연결이 약함.

## 4.2 개선안 (와이어프레임 급)

```
┌───────────────── 솥단지 — 조합 ──────────────── [✕] ┐
│  [ 슬롯0 ]  +  [ 슬롯1 ]   =   [ 결과 미리보기 ]      │
│   (재료)      (재료)          실루엣/? or 확정 결과    │  ← 4.3-A 실시간 미리보기
│                  (중앙 솥 그래픽 / fly-in 주스 유지)   │
│  [ 비우기 ]            [ 조합 ]                        │
│  상태/flavor 라인                                     │
│  힌트 ●●●○○  ▸힌트 보기(N)                            │
│ ─────────────────────────────────────────────────── │
│  재료 선택  [🔍필터: 전체|물|흙|광물|…] [최근 ⟳]     │  ← 4.3-B 필터 / 4.3-C 최근
│  [아이콘 재료 x3][아이콘 재료 x1] … (스크롤)          │
│  ─ 최근 조합 ─  [I1+I7 ⟳][I2+I5 ⟳] …                 │  ← 4.3-C 원탭 재조합
│  도감: 87/220 발견   [도감 열기 R]                    │  ← 4.3-D 도감 연결
└──────────────────────────────────────────────────────┘
```

## 4.3 개선 항목

- **A. 실시간 결과 미리보기(§2 연동).** 슬롯 2개가 채워지면 조합 전에 결과 슬롯을 갱신:
  - 이미 발견한 레시피 → 결과 아이콘/이름 확정 표시(재조합 확신).
  - 미발견이지만 유효 레시피 → **결과 실루엣 + 시적 조각**(§2.2 도감 힌트 단계에 맞춰).
  - 유효 레시피 없음 → "?" + (실패작 매핑 있으면) 회색 힌트.
  - 구현: 슬롯 변경 시 `Fusion.peek(a,b)`(신규 순수 조회, 소비 없음) 호출 → 결과 슬롯 렌더. 로직 무변경(실 조합은 여전히 조합 버튼).
- **B. 재료 후보 필터.** 스트립 상단에 카테고리 칩(물/흙/광물/식물/기계…) + (선택)텍스트 검색. items.json의 속성/레이어로 필터. 슬롯 1개 채운 상태면 "이 재료와 조합 가능한 것" 필터 옵션(추리 보조, 난이도 옵션화).
- **C. 최근/반복 조합 편의.** 성공 조합을 최근 N개(예 6) 큐로 보관 → "최근 조합" 행에 [재료A+재료B ⟳] 원탭 버튼. 누르면 두 슬롯 자동 채움(재고 있을 때). 반복 크래프팅(Whisper 벌이 등) 대폭 단축.
- **D. 도감 연결.** 하단에 "도감 N/M 발견" + [도감 열기 R] 바로가기. 발견/미발견 카운트가 조합 동기 부여.

## 4.4 구현 노트

- **로직 불변 원칙 유지**(현 코드의 강한 계약): `Fusion.fuse`는 그대로. 미리보기는 **소비 없는 `Fusion.peek`** 신설로만.
- 필터/최근/도감 연결은 **기존 `outer` VBox에 행 추가** — 뷰포트 클램프(`_clamp_to_viewport`)가 이미 스크롤 처리하므로 세로 증가 흡수.
- 최근 조합 큐는 세이브에 `recent_fusions:[[a,b],…]` 추가(옵션, 세션 한정도 가능).
- 모바일: 최근 조합/필터 칩 ≥44px 터치 타겟(현 스트립 규약 계승).
- §2·§4는 **결과 미리보기 지점에서 한 몸** — 함께 구현 권장(§6 GP 카드에서 인접 배치).

# §3. 게이트 미니 퍼즐화 (레이어당 1개 승격)

## 3.1 현행 게이트 모델 (l2~l5_gate_controller.gd 실측)

각 레이어 게이트는 **"조합물 장착(mount) → 효과 발화"** 모델이다: 크래프트한 아이템(전지 D64, 퓨즈 D66, 룬 등)을 스폰된 오브젝트(배전반/발전기/룬 다리)에 `item_used_on_object`로 장착 → `power_node_energized(node_id)` 등이 발화 → 게이트 개방. 컨트롤러가 `GameState.item_used_on_object`를 듣고 올바른 노드 효과를 발화. idempotent(로드 시 재적용).

**즉 게이트는 이미 "장착 지점"이 있다.** 미니 퍼즐은 그 장착 순간에 **1화면 UI 레이어**를 얹는다 — 기존 장착 로직은 퍼즐 성공 시 그대로 호출(저비용).

## 3.2 설계 원칙

- **레이어당 정확히 1개** 게이트만 승격(나머지는 현행 장착형 유지 — 리듬 완급).
- **1화면 미니 퍼즐**: 장착 상호작용 → 퍼즐 모달 팝업 → 성공 시 기존 개방 로직 호출, 실패 시 재시도.
- **기존 장착 위 UI 레이어만**: 게이트 상태머신/개방 로직 무변경. 퍼즐은 성공/실패만 반환.
- **접근성 필수**: 모든 퍼즐에 **스킵 버튼**("건너뛰기 — 그냥 장착"). 퍼즐은 재미 가산이지 진행 차단이 아니다(오너 진행 막힘 트라우마 회피). 스킵 시 기존 장착과 동일 결과.

## 3.3 레이어별 미니 퍼즐 스케치

### L2 배전반 = 퓨즈 순서 맞추기
- **화면**: 배전반에 빈 퓨즈 슬롯 4개 + 색 표시등. 목표 순서(예 남색→시안→남색→시안)가 상단에 깜빡임 → 플레이어가 퓨즈를 그 순서로 꽂기.
- **조작**: 퓨즈 클릭/탭 → 다음 빈 슬롯에 장착(색 순환 or 재고 소비).
- **실패**: 순서 틀리면 해당 슬롯 빨강 점멸 + 초기화(감점 없음, 무한 재시도).
- **스킵**: "그냥 전력 연결" → 순서 무시하고 개방.
- **연결**: 성공 → 기존 `power_node_energized("bridge")` 호출.

### L3 톱니 = 회전·크기 맞물림 배치 퍼즐
- **화면**: 축 3~4개에 크고 작은 톱니를 배치 → 좌측 구동 톱니에서 우측 출력 톱니까지 **맞물려 회전이 전달**되게.
- **조작**: 톱니를 축에 드래그/탭 배치. 인접 톱니가 맞물리면 회전 애니(피드백).
- **실패**: 맞물림 끊기면 출력 톱니 정지 → 개방 안 됨. 배치 자유 조정.
- **스킵**: "수동으로 태엽 감기".
- **연결**: 출력 톱니 회전 성립 → 기존 L3 게이트 개방 노드 발화.

### L4 룬 다리 = 룬 순서 점등
- **화면**: 다리 위 룬 5~6개. 상단에 목표 점등 순서가 잠깐 빛났다 사라짐(기억) → 그 순서로 룬을 밟기/터치.
- **조작**: 룬 클릭/탭 → 점등. 순서대로면 다음 룬 활성.
- **실패**: 틀린 룬 → 전체 소등 후 순서 재표시(패널티 없음).
- **스킵**: "룬을 억지로 잇기".
- **연결**: 전 룬 점등 → 기존 L4 룬 다리 개방.

### L5 성가 = 음 순서 기억
- **화면**: 파이프 오르간/종 4~5개(각 음+색). 성가 한 소절이 순차 연주(음+발광) → 플레이어가 재현.
- **조작**: 종 클릭/탭 → 음 연주+발광. 사운드 파이프라인 재사용(기존 18종+).
- **실패**: 틀린 음 → 소절 재생 후 재시도. 소절 길이 점증(2→3→4음) 난이도.
- **스킵**: "침묵으로 기도" — 순서 없이 개방(성화 톤).
- **연결**: 소절 완주 → 기존 L5 성가 게이트 개방.

## 3.4 구현 노트

- **공통 베이스** `GatePuzzle`(CanvasLayer 모달): `open(on_success, on_skip)` 콜백 계약. 4종 퍼즐이 이 베이스를 상속(퓨즈순서/톱니맞물림/룬점등/음순서). fusion_ui 모달 패턴(뷰포트 클램프·push_modal) 재사용.
- **장착 지점 후킹**: 각 `lN_gate_controller`의 해당 게이트 장착 핸들러에서, 실제 개방 발화 직전에 `GatePuzzle.open(...)`을 끼운다. 성공/스킵 콜백이 **기존 개방 함수를 그대로 호출** — 개방 로직 0줄 변경.
- **헤드리스/하네스**: 각 퍼즐에 `solve_for_test()`(정답 즉시 입력)로 하네스가 게이트 진행 검증. 스킵 경로도 테스트.
- **접근성**: 스킵은 항상 노출(옵션 토글 아님). 색맹 대응 — 색+아이콘/숫자 병기(퓨즈 색에 숫자, 룬에 문양).
- **비용**: 퍼즐 4종 UI + 훅 4곳. 게이트 로직 무변경이라 회귀 위험 낮음. 레이어별 **병렬 착수 가능**(서로 독립).
